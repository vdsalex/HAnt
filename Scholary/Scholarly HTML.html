1. Abstract

Hazard Smart Assistant is a web application whose purpose is to provide real-time information about currently occurring natural disasters.

2. Introduction

Hazard Smart Assistant is an online tool created for collecting and sharing information regarding natural disasters, such as: fire, floods, earthquakes and tornadoes. We consider as relevant any of the following data about a natural disaster: the nature of the disaster, the location of the disaster, the location of the disaster, areas around the hazard and the time when the hazard was reported. The collected data is then highlighted in an interactive map through map markers, which offer all of the information regarding a hazard: the location shown by the marker is the location of the disaster, and, upon clicking any marker, a dialog will appear above it showing: the type of the disaster, weather conditions and the time when it was reported, as well as a button that will redirect to a new page containing more details about the disaster, such as: degree of danger, number of victims, impacted area(s) and more.

This data is then sent to the relevant authorities and waits for updates from them, while logging further updates. All these updates will be shown in the map page, while updates regarding any particular hazard will be available in the details page of said hazard.

3. Structure

Being a web application, Hazard Smart Assistant has 3 parts: Front-end, Back-end and storage.

3.1 Front-end

The front-end component contains 3 pages: the main page, the map page and the details page.

The main page has the application logo and acts as a "loading screen", as the logo will spin while there are no active hazards.

The map page presents a marked Google Map, along a textbox. Whenever a hazard becomes active (through a reporting process), the user will be automatically redirected to this page, where hazard locations are shown via Google Map markers. Upon clicking a marker, more data regarding it's corresponding hazard will be shown, as well as a "details" button for the history of said hazard. The textbox contains the history of all the hazards as short descriptions like "X hazard happened at Y location", and it will be dynamically updated whenever a new hazard becomes active.

The details page contains all the data related to a particular hazard, such as: hazard type, initial location, spread, hazard history, current status (active / inactive), date and time of detection. To access this page, the user has to select any marker on the map page and then click the "details" button.

3.2 Back-end

The back-end component contains the functional side of the project, having as the main modules the detection, marking, forwarding and tracking of a hazard.

Considering that most areas are equipped with sensors for fire, flood, earthquake and wind speed, a hazard is detected whenever a sensor sends a message or signal reporting the appearance of a hazard type. If no hazard is detected for a considerable amount of time or all hazards have been resolved, the user will be redirected to the main page while being given the possibility to access the map page.

Upon detecting a hazard, it's location is marked on the map by using the gps sensor connected to the reportng sensor. From this point on, the hazard is considered as "active"

The forwarding module contains two functions: announcement function and instruction function. The announcement function sends the hazard data to the authorities and further updates this data until the hazard is resolved, at which point the hazard will be considered inactive. The authorities will first receive the hazard type and location, while being able to send back messages like: "intervention", "evacuating" and "finished". When the "finished" message is received, the hazard is considered to be resolved. The instruction function will send evacuation instructions if the location of the hazard is a building. These instructions will prioritize the evacuation of superior floors, as well as the evacuation of people with high mobility abilities or good health in this order.

Finally, the tracking module will record all updates regarding a hazard, from it's detection, to the actions taken(evacuation, announcement of authorities, fighting the hazard - when possible), to it's resolving and dynamically update the textbox in the map page, as well as the details page for the specific hazard, whenever a new action is taken regarding said hazard. This is done in order to allow following a singular hazard when multiple hazards take place at the same time, while also showing how many hazards are active at a given moment.

3.3 Storage

The information regarding hazards and buildings (currently only hospitals) will be stored in a knowledge database using the RDF format. Queries for this database will be done using a SPARQL endpoint. We will be using RdfStore for working wth rdf`s also for storage rdf data.

4. Technologies

The technologies used for this project are: Angular 7 for the front-end component, Node.js for the back-end component, RdfStore for the knowledge storage and a SPARQL endpoint. We picked RdfStore because it is compatible with SPARQL 1.0 and almost total coverage for 1.1 also because we can mange RDF storage or queries. For image conversion we used an online svg convertor. We will use express to build our API in node.js, which will work with http request and http response messages.

4.1 Angular 7

Angular 7 is the latest version of the angular framework, which is a TypeScript-based open-source front-end web application platform led by the Angular Team at Google and by a community of individuals and corporations.

4.2 Node.js

Node.js is an open-source, cross-platform JavaScript run-time environment that executes JavaScript code outside of a browser. We will use node.js with express and RdfStore to build our API.

4.5 RDF

The Resource Description Framework is a family of World Wide Web Consortium specifications originally designed as a metadata data model. The RDF data model is similar to classical conceptual modeling approaches (such as entity–relationship or class diagrams). It is based on the idea of making statements about resources (in particular web resources) in expressions of the form subject–predicate–object, known as triples. The subject denotes the resource, and the predicate denotes traits or aspects of the resource, and expresses a relationship between the subject and the object.

4.6 SPARQL

SPARQL is an RDF query language—that is, a semantic query language for databases—able to retrieve and manipulate data stored in Resource Description Framework format. We wll need SPARQL 1.0 becouse we will make just get requests we don`t need to update our RDF storage.
